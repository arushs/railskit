# Deployment Guide

RailsKit ships with production-ready configs for Render, Fly.io, Docker, and Kamal.

---

## Option 1: Render (Recommended)

A `render.yaml` Blueprint is included. One-click deploy.

### Steps

1. Push your repo to GitHub
2. Go to [Render Dashboard](https://dashboard.render.com) → **New Blueprint Instance**
3. Connect your repo — Render reads `render.yaml` automatically
4. Set environment variables (Stripe keys, AI keys, etc.)
5. Deploy

### What `render.yaml` Creates

| Service | Type | Description |
|---|---|---|
| `railskit-db` | PostgreSQL | Free tier database |
| `railskit-api` | Web Service (Docker) | Rails API at `/up` health check |
| `railskit-web` | Static Site | React app built with `npm run build` |

### Environment Variables to Set

After the blueprint creates the services, set these manually in Render:

- `CORS_ORIGINS` — your frontend URL (e.g., `https://railskit-web.onrender.com`)
- `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`
- `OPENAI_API_KEY` (or your chosen AI provider key)
- `DEVISE_JWT_SECRET_KEY` — generate with `rails secret`
- `VITE_API_URL` — your API URL (set on the static site service)

`SECRET_KEY_BASE` and `DATABASE_URL` are auto-generated by Render.

## Option 2: Fly.io

A `fly.toml` is included.

### Steps

```bash
# First time: create the app
fly launch --copy-config --no-deploy

# Create the database
fly postgres create --name railskit-db
fly postgres attach railskit-db

# Set secrets
fly secrets set \
  SECRET_KEY_BASE=$(rails secret) \
  DEVISE_JWT_SECRET_KEY=$(rails secret) \
  STRIPE_SECRET_KEY=sk_live_... \
  STRIPE_WEBHOOK_SECRET=whsec_... \
  OPENAI_API_KEY=sk-... \
  CORS_ORIGINS=https://your-frontend.com

# Deploy
fly deploy
```

### What `fly.toml` Configures

- **Region:** `iad` (US East) — change to your preferred region
- **VM:** shared-cpu-1x, 512MB RAM
- **Health check:** `GET /up` every 15s
- **Release command:** `bin/rails db:prepare` (runs migrations on deploy)
- **Deploy strategy:** rolling
- **Auto-scaling:** suspend idle machines, auto-start on traffic

### Frontend

Deploy the React app separately (Vercel, Cloudflare Pages, or Render Static):

```bash
cd web
npm run build
# Upload dist/ to your static hosting
```

Set `VITE_API_URL` to your Fly.io API URL at build time.

## Option 3: Docker

`docker-compose.yml` runs everything locally or on any Docker host.

### Development

```bash
docker compose up
```

This starts:
- **PostgreSQL** (port 5432)
- **Redis** (port 6379)
- **Rails API** (port 3000)
- **Vite dev server** (port 5173)

### Production

Build the production image:

```bash
docker build -f Dockerfile.production -t railskit-api .
```

Run with your production env vars:

```bash
docker run -p 3000:3000 \
  -e DATABASE_URL=postgres://... \
  -e RAILS_ENV=production \
  -e SECRET_KEY_BASE=... \
  -e STRIPE_SECRET_KEY=... \
  railskit-api
```

For the frontend, build the static files and serve from any CDN/static host:

```bash
cd web && npm run build
# Serve dist/ from Nginx, Cloudflare Pages, Vercel, etc.
```

## Option 4: Kamal

`api/config/deploy.yml` is pre-configured for Kamal (Rails' official deploy tool).

### Steps

```bash
cd api

# Edit config/deploy.yml:
# - Set your server IP
# - Set your container registry
# - Set your domain

# Deploy
bin/kamal setup    # first time
bin/kamal deploy   # subsequent deploys
```

### Kamal Aliases

```bash
bin/kamal console   # Rails console on server
bin/kamal shell     # SSH into container
bin/kamal logs      # Tail logs
bin/kamal dbc       # Database console
```

## Production Checklist

### Required

- [ ] Set `SECRET_KEY_BASE` (generate with `rails secret`)
- [ ] Set `DEVISE_JWT_SECRET_KEY` (generate with `rails secret`)
- [ ] Set `DATABASE_URL` to your production PostgreSQL
- [ ] Set `CORS_ORIGINS` to your frontend domain
- [ ] Set `FRONTEND_URL` to your frontend domain (for OAuth redirects)
- [ ] Configure Stripe webhook endpoint in Stripe Dashboard → point to `https://your-api.com/api/webhooks/stripe`
- [ ] Set `STRIPE_WEBHOOK_SECRET` to the production webhook signing secret
- [ ] Run `bin/rails db:prepare` (creates DB + runs migrations)
- [ ] Run `bin/rails db:seed` (seeds plans)

### Recommended

- [ ] Enable SSL (`config.force_ssl = true` in production.rb)
- [ ] Set up a custom domain
- [ ] Configure Google OAuth redirect URIs for production domain
- [ ] Set up error monitoring (Sentry, Honeybadger, etc.)
- [ ] Set up log aggregation

### Solid Queue in Production

By default, Solid Queue runs inside Puma (`SOLID_QUEUE_IN_PUMA=true`). For higher job throughput, run a dedicated worker:

```bash
# In a separate process/container:
SOLID_QUEUE_IN_PUMA=false bin/rails solid_queue:start
```
